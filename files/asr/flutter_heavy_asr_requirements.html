方案 B · 客户端适配器实时 ASR 需求规格

本文定义方案 B 的客户端实现规格：Flutter 库持有统一会话内核和多模型适配器。业务层在调用语音能力前先调用 SDK 的 init，并传入 appName 与可选 userToken；SDK 再请求配置接口获取模型配置并决定是否切换。第一版以 Qwen3-ASR-Flash-Realtime 为首个落地模型。

1. 范围与前提

项约束
平台范围仅支持 Android 和 iOS。
识别范围仅支持实时流识别，不承诺离线能力，不承诺断线无缝续识别。
配置来源业务层调用 SDK 的 init 后，由 SDK 请求后端配置接口获取 apiKey、模型、endpoint 等运行期配置；请求必须至少带 appName，必要时可带环境、渠道等参数。
用户维度是否附带 userToken 由业务需求决定；若后端只按 appName 维度下发配置，userToken 可以为空。
权限边界麦克风权限检查与申请由业务层完成；SDK 不主动申请权限，只在权限未就绪导致录音初始化失败时返回明确错误。
切换策略配置相同则不切换；配置不同才切换；切换只影响新建会话，不影响进行中的会话。
并发约束同一页面不允许并行多个识别会话。
多厂商策略允许多供应商并存，但每次会话只绑定一个模型适配器。
错误与事件必须保留供应商原始错误码、原始消息和原始事件直返。
第一版目标先定统一架构并打通 Qwen3-ASR-Flash-Realtime。
本方案中，“模型能力”由适配器实现层声明和处理，而不是要求业务方在统一配置里传递厂商特有能力参数。统一配置只承载路由和必要连接信息。

2. 总体架构

方案 B 的核心不是一个统一后端协议，而是一个统一会话内核，加上一组按模型实现的适配器。

模块职责说明
AsrFacade业务唯一入口向业务暴露 init、createSession、release；识别控制通过 AsrSession 执行 start、stop、cancel。
ConfigResolver拉取配置并生成快照在 init 阶段携带 appName 和可选 userToken 请求后端配置，输出可比较的 AsrConfigSnapshot。
ConfigComparator决定是否切换比较当前生效配置与新配置是否一致。
PageSessionGuard同页单会话守卫禁止同一页面同时存在多个活动识别会话。
SessionController维护主状态机串行化命令、约束生命周期、管理终态收口。
AudioSource音频采集和分帧在业务层已完成权限准备前提下处理音频采集、帧切分与资源释放。
ModelAdapter统一模型接口每个新增模型必须实现该接口。
QwenRealtimeAdapter首个模型实现第一版具体模型适配器。
EventMapper统一事件映射把适配器事件转成统一事件，同时保留 raw 通道。
RawEventChannel原始事件直返原始事件保真输出，不直接驱动主状态机。

3. 配置获取与切换规则

配置快照字段
  • appName
  • environment
  • channel
  • providerId
  • modelId
  • apiKey
  • endpoint
  • protocolVersion
  • extra
  • resolvedAt
  • sessionBindingId
配置请求上下文
  • 每次调用 init 时只拉取一次配置。
  • 请求至少包含 appName，必要时可带 appVersion、channel、environment。
  • userToken 作为可选字段，仅在需要按用户维度下发配置时传入。
  • 配置一致则复用当前适配器和路由配置，不重建模型上下文。
场景要求
进入页面拿到相同配置直接复用，不重建适配器，不影响已有会话状态。
进入页面拿到不同配置，且无活动会话销毁旧适配器上下文，切换到新配置和新适配器。
进入页面拿到不同配置，且有活动会话拒绝立即切换；必须等待 stop / cancel / failed 结束后，后续新会话再使用新配置。
配置过期或无效当前会话若未启动则拒绝进入 Ready；若已启动，按适配器错误收口进入 Failed。
配置生命周期必须独立于识别会话生命周期。ConfigComparator 只比较 appName、environment、channel、providerId、modelId、apiKey、endpoint、protocolVersion 和 extra 等路由字段；userToken 只有在后端明确按用户维度返回配置时才参与配置判定。resolvedAt 与 sessionBindingId 仅用于运行期观测和会话绑定，不参与是否切换的判定。一次会话绑定一次配置快照；正在运行的会话不会被再次 init 而隐式切换。配置采用长期存储还是短期存储不由 SDK 接口强制规定，但 release 后必须清理运行期持有数据。

4. 统一模型接口与适配器

统一模型接口不是统一配置，而是统一调用契约。厂商能力和语义差异由适配器实现层承担。

接口对象职责实现要求
ModelAdapter模型适配总接口定义 start、sendAudio、stop、cancel、dispose、capabilities、eventStream、rawEventStream。
AdapterCapabilities适配器能力声明由实现层声明 supportsPartial、supportsTimestamp、supportsNoSpeechCompleted 等，不要求业务配置显式携带。
AdapterConfigValidator配置校验验证当前快照是否足以启动该模型。
AdapterEventMapper事件归一化把原始事件映射成 RecognitionPartial、RecognitionFinal、SessionCompleted、SessionCancelled、SessionFailed。
AdapterErrorMapper错误归一化输出最小统一错误分类，同时保留原始错误码和原始载荷。
统一模型接口不能退化成“所有供应商都有同样字段”的最小公分母。真正需要统一的是生命周期和事件契约；厂商能力差异应由适配器声明，而不是强压到统一配置里。

5. 会话与状态机

状态说明允许动作退出条件
Idle尚未 init，或已经 release。init进入 ConfigLoading 或 Failed
ConfigLoading正在获取并比较配置。cancel进入 Ready、Failed 或 Idle
Ready配置与适配器已确定，可创建会话。createSession, release进入 Preparing 或 Idle
Preparing校验业务层是否已完成权限前置条件、初始化音频、建立模型连接。cancel进入 Streaming、Cancelled、Failed 或 Idle
Streaming持续送音频并接收 partial 和 raw 事件。stop, cancel进入 Stopping、Cancelled、Failed 或 Idle
Stopping停止送音，等待 final / completed 收尾。cancel进入 Completed、Cancelled、Failed 或 Idle
Completed正常完成终态。release进入 Idle
Cancelled主动取消终态。release进入 Idle
Failed异常结束终态。release进入 Idle
同一页面不允许并行多个识别会话。Ready、Preparing、Streaming、Stopping 任一阶段再次 start，都必须被 PageSessionGuard 拒绝。会话进入 Completed、Cancelled、Failed 任一终态后，音频采集、连接和会话级监听都必须立即释放；release 负责清理 SDK 级运行期配置和可复用状态。

6. 事件与错误模型

统一事件通道
  • SessionStateChanged
  • RecognitionPartial
  • RecognitionFinal
  • SessionCompleted
  • SessionCancelled
  • SessionFailed
原始事件通道
  • ProviderRawEvent
  • ProviderRawError
  • RawPayload
  • RequestId
  • SessionId
统一错误最小分类
  • ConfigError
  • PermissionError
  • AudioError
  • TransportError
  • ProviderError
  • StateError
规则说明
统一状态机唯一权威业务侧主逻辑必须依赖统一状态和统一终态事件，而不是直接消费原始事件判断业务流程。
权限职责边界业务层负责权限检查与申请；SDK 不主动申请权限，只在录音初始化失败时以 PermissionError 或 Failed 终态返回。
原始事件保真输出适配器必须原样保留供应商原始 errorCode、errorMessage、payload、requestId。
终态唯一性一次会话只能有一个统一终态事件：Completed、Cancelled 或 Failed 三者只能出现一个。
stop 收尾顺序正常 stop 路径下，必须先输出 RecognitionFinal，再输出 SessionCompleted；若本次会话没有可提交文本，则允许仅输出带 no-speech 原因的 SessionCompleted。
cancel 规则cancel 后只允许输出 SessionCancelled，不再产出 final 或 completed。
终态资源释放会话进入 Completed、Cancelled、Failed 后，必须立即释放音频、传输连接和会话级监听；release 仅负责继续清理 SDK 级运行期配置与内部缓存。

7. 安全前提与边界

项要求
密钥管理apiKey 仍由后端统一管理、下发、吊销和轮换；Flutter 端只在 init 成功后的运行期消费配置，不应把密钥长期持久化到业务存储。release 后必须清理运行期持有配置。
风险定性“后端向 Flutter 返回长期 apiKey”属于当前业务约束下的高风险折中，只能视为阶段性实现前提，不能视为长期推荐安全基线。
配置边界配置接口至少带 appName，必要时可带环境、渠道等上下文；userToken 仅在需要按用户维度下发配置时作为可选字段传入。
日志要求诊断日志和原始事件默认脱敏，不直接输出完整 apiKey、完整鉴权头和可复用敏感载荷。
方案 B 的安全语义必须同时成立：一方面，后端统一管理密钥使泄露后可以快速吊销和轮换；另一方面，长期 apiKey 进入客户端仍会扩大暴露面，因此不能把“可吊销”误解成“低风险”。

8. 第一版首模型要求

Qwen3-ASR-Flash-Realtime 接入要求
  • 作为首个真实 ModelAdapter 落地。
  • 必须走统一会话内核，不允许业务直接调用模型私有 API。
  • 必须产出 raw event 和统一事件双通道。
  • 必须验证 stop / cancel / failed 三条主收口路径。
防止首模型绑死架构
  • 先定义 provider-neutral contract，再实现 Qwen 适配器。
  • 至少保留一个 FakeAdapter 作为契约测试基线。
  • 公共接口不得泄漏 Qwen 私有字段。
  • 公共状态机不得依赖 Qwen 私有事件顺序。

9. 测试与验收

测试层级覆盖重点验收要点
单元测试配置比较、PageSessionGuard、状态机、事件映射、错误映射。配置相同不切换、不同才切换；终态唯一；重复 start 被拒绝。
集成测试Qwen 适配器、init / release、音频采集、实时发送、stop / cancel / failed 收口，以及不同 appName 上下文下的配置返回。一条完整链路可稳定输出 partial、final、completed，cancel 不再输出 final。
真机测试业务层权限编排、前后台切换、来电打断、弱网、页面退出重进。页面重进后配置逻辑稳定，活动会话不会被热切换。
必须验证的切换场景
  • 配置相同复用旧配置。
  • 不同 appName 上下文返回不同配置时能正确切换。
  • 配置变化但页面无活动会话时完成切换。
  • 配置变化且有活动会话时拒绝热切换。
必须验证的会话场景
  • 正常 start 到 stop 闭环。
  • cancel 立即结束。
  • 网络断开进入 Failed。
  • userToken 缺失时，若后端配置策略只按 appName 维度工作，仍能完成正常闭环。
必须验证的原始事件场景
  • 原始错误码和消息未丢失。
  • 统一事件和 raw 事件顺序可预测。
  • 终态不重复派发。
