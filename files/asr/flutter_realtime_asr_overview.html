方案 A · 后端中转实时 ASR 概览

这份文档只回答方案 A 中 Flutter 客户端库需要做什么。业务层在调用语音能力前，需要先调用 SDK 的 init，并传入 appName 与可选 userToken，由 SDK 请求配置接口获取当前运行期配置；若需要释放这些配置和资源，则调用 release。若需要完整规格、状态机、错误策略和验收口径，请进入方案 A 需求规格。

平台：Android / iOS 目标：实时流识别闭环 不承诺：离线能力、无缝续识别

库需要做什么

方案 A 中，Flutter 端的职责要足够清晰，避免和后端职责混在一起。这个库的目标不是直连所有供应商，而是作为统一的实时 ASR 客户端库，让多个 Flutter App 可以直接依赖接入，并由 SDK 根据 init 传入的 appName 与可选 userToken 请求后端决定实际路由与配置。

1. 提供统一接入入口

  • 对业务层暴露 init、createSession、release 等基础入口。
  • init 负责请求配置接口并持有当前运行期配置。
  • 让业务方通过统一方式监听识别结果和状态变化。

2. 负责本地音频采集

  • 在业务层已完成权限准备前提下启动音频采集。
  • 启动和停止实时录音采集。
  • 把本地音频收敛成稳定的分帧输入格式。

3. 负责实时会话连接

  • 在 init 已拿到运行期配置后，与后端建立 WebSocket 流式连接。
  • 持续发送音频帧并接收识别结果。
  • 处理握手、ready、收尾和关闭流程。

4. 负责客户端状态机

  • 管理 Idle、Preparing、Streaming、Stopping、Completed、Cancelled、Failed 等状态。
  • 区分正常结束、主动取消和异常失败。
  • 保证状态变化对业务层可监听、可判定。

5. 输出统一事件

  • 输出 partial result 和 final result。
  • 输出 completed、cancelled、failed 等终态事件。
  • 保留 provider 原始诊断信息作为扩展事件。

6. 预留扩展能力

  • 支持 capability 声明，例如 partial、timestamp、speaker 等。
  • 支持 metadata 或 extra 透传扩展字段。
  • 允许未来新增模型时主要通过后端适配，不破坏客户端主接口。

安全边界

Flutter 端不在业务侧硬编码供应商 API Key。业务层通过 init 传入 appName 与可选 userToken 后，由 SDK 请求配置接口获取运行期配置；是否长期存储由实现决定，但 release 后必须清理运行期持有数据。

稳定性边界

本版本不承诺断线后续接原会话，连接异常直接结束当前会话并释放资源。

接入方式

其他 Flutter 项目通过依赖方式直接引入库，而不是复制源码到业务工程中。
