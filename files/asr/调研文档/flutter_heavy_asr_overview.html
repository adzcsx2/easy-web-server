方案 B · 客户端适配器实时 ASR 概览

这是一套客户端主导的实时 ASR 方案。方案 B 中，业务层在调用语音能力前先调用 SDK 的 init，并传入 appName 与可选 userToken；SDK 再请求配置接口获取模型配置和 apiKey，并据此选择具体模型适配器。配置采用长期存储还是短期存储不是 SDK 接口强制要求，但 release 后必须释放这些运行期配置与资源。麦克风权限检查与申请由业务层承担，SDK 不主动申请权限。第一版首个模型为 Qwen3-ASR-Flash-Realtime。

平台：Android / iOS 目标：实时流识别闭环 约束：同页单会话，页面级拉配置 高风险前提：长期 apiKey
当前方案由后端统一管理 apiKey 并下发配置给 Flutter 端。密钥由后端管控，泄露时可即时吊销更换，但“后端下发长期 apiKey”仍属于当前业务约束下的高风险折中，不应视为长期推荐安全基线。

库需要做什么

方案 B 中，Flutter 库不再只是一个统一后端协议客户端，而是一个拥有统一会话内核和多模型适配器的客户端识别库。业务页面只依赖统一接口，不直接依赖厂商协议。

1. 通过 init 拉取配置

  • 业务层调用 init，并传入 appName 与可选 userToken。
  • SDK 在 init 阶段向后端请求当前 App 对应的模型配置。
  • 请求中还可带环境、渠道等上下文；是否带 userToken 取决于业务是否需要按用户维度下发配置。
  • 比较当前配置与已生效配置是否一致，只比较影响路由和连接的字段。
  • 配置相同则复用，不同才切换适配器。

2. 提供统一模型接口

  • 新增模型必须实现统一适配器接口。
  • 业务调用层只依赖统一会话接口。
  • 厂商差异在接口实现层收敛，不进入业务主调用面。

3. 管理单页单会话

  • 同一页面任意时刻只允许一个识别会话。
  • 重复 start 必须被拒绝或显式报错。
  • 配置变化不能热切换正在运行的会话。

4. 负责实时识别闭环

  • 在业务层已完成权限准备前提下处理音频采集与音频分帧。
  • 建立模型实时连接并发送音频流。
  • 输出 partial、final、completed、cancelled、failed。

5. 保留原始错误和事件

  • 统一状态机负责主业务语义。
  • 原始错误码、原始消息、原始事件直返。
  • 原始通道用于诊断，不直接驱动主状态机。

6. 第一版先打通一个模型

  • 首个模型为 Qwen3-ASR-Flash-Realtime。
  • 先把统一架构和首个适配器落稳。
  • 后续新增模型按同一适配器契约迭代接入。

适用边界

仅支持 Android 和 iOS，只做实时流识别，不承诺离线能力，不承诺断线无缝续识别。

切换边界

运行时动态切换按页面级进行。每次 init 拉一次配置，同配置复用，不同配置才切换，且只影响新建会话。

实现边界

模型能力由适配器实现层声明和处理，不要求业务在统一配置中传递厂商特有能力开关。
