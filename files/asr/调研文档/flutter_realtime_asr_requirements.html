1. 文档目的与定位

本规格文档用于定义方案 A 中 Flutter 客户端库的边界、职责、公开能力、事件模型、状态机行为与验收口径。本文档是客户端库需求规格，不是后端网关方案文档，也不是某一家供应商的协议文档。

库定位
  • 作为其他 Flutter 项目可直接依赖引入的客户端库交付。
  • 同一个 SDK 应支持被多个 App 复用，由 SDK 根据 init 传入的 appName 与可选 userToken 请求后端解析对应会话路由与策略。
  • 默认面向实时麦克风采集场景，也允许扩展为外部音频帧输入。
  • 客户端面向自有后端实时 ASR 会话协议，而不是直接绑定具体厂商协议。
目标读者
  • Flutter SDK 开发人员。
  • 移动端业务接入方。
  • 后端协议设计和联调人员。
  • 测试与发布验收人员。

2. 范围与非目标

本章节用于消除范围歧义，确保 Flutter 库只承担合理的客户端职责，不把后端和供应商差异转嫁到 SDK 公开接口上，也不把业务权限编排固化到 SDK 内部。

包含范围
  • 在业务层已完成麦克风权限准备的前提下，完成 Android、iOS 平台的实时采集。
  • 与后端建立 WebSocket 实时流会话并持续发送音频帧。
  • 接收中间识别结果、最终结果、状态事件、诊断事件。
  • 客户端识别状态机与资源生命周期管理。
  • 支持不同供应商能力经由后端协议统一接入。
明确非目标
  • 不承诺断网后的无缝续识别。
  • 不承诺离线识别或本地模型识别。
  • 不支持历史文件上传转写与批量转写。
  • 不在 Flutter 库内管理供应商 API Key、模型切换与配额路由。
  • 不承诺多供应商结果一致性或特性完全对齐。
约束原则：Flutter 库只连接自有后端实时协议，后端负责吸收不同厂商的鉴权、路由、协议差异和动态切换逻辑。客户端保留诊断原文，但不把厂商协议直接暴露成主接口。

3. 设计原则

原则 约束要求
安全优先 客户端不得在业务侧硬编码供应商密钥；业务层通过 init 传入 appName 与可选 userToken 后，由 SDK 请求配置接口获取运行期配置。配置持久化周期可按实现决定，但 release 后必须清理运行期持有的敏感数据；诊断日志默认脱敏。
状态可判定 所有公开行为必须落在明确状态机上；终止态后不允许隐式复活同一会话。
基础能力统一 统一会话控制、结果事件、状态事件；供应商差异通过扩展字段与能力声明承载。
按应用维度解析配置 SDK 必须对业务层暴露 init，并允许传入 appName、环境、渠道等上下文以及可选 userToken，由 SDK 请求配置接口后获取当前会话所需配置。
失败即收口 网络断开、握手失败、录音失败等异常统一进入失败终态并释放资源，不伪装为无缝续流。
单文件可接入 其他 Flutter 项目应能通过依赖方式直接接入，无需复制源码或修改内部实现。

4. 模块与分层

推荐在单一 SDK 内采用核心层与平台层分离的内部结构，既便于发布，也便于测试和后续扩展。

层级 职责 是否公开
Public Facade 向业务方暴露 init、创建会话、开始、停止、取消、release、监听事件的统一入口。 是
Session Controller 维护状态机、串行化命令、统一管理录音与传输资源。 是
Audio Source Layer 封装音频采集、分帧、采样率与格式收敛；默认在业务层已完成权限准备的前提下运行。 内部为主
Transport / Adapter Layer 封装与后端的 WebSocket 流式会话、编解码、心跳、超时、背压与协议版本。 内部为主
Event Model 定义 partial、final、stateChanged、warning、error、providerRaw 等事件。 是
推荐发布形式
  • asr_flutter：唯一对外交付的 Flutter SDK，业务方只依赖这一个库。
  • SDK 内部按模型、状态机、传输层、音频采集与测试夹具分层组织。
  • 内部模块拆分不应转化为业务侧多包依赖。
为何不做多厂商直连 SDK
  • 避免客户端直接依赖供应商协议与错误语义。
  • 避免模型切换时同步升级 App。
  • 保持公开 API 稳定，新增供应商主要在后端完成。

5. 功能需求

编号 功能 要求
FR-01 初始化与配置获取 库必须对业务层暴露 init(initParams)。业务层至少传入 appName，必要时可附带 appVersion、channel、environment 和可选 userToken；SDK 在 init 阶段请求配置接口，并持有当前运行期配置供后续语音能力使用。
FR-02 会话创建 createSession 只能在 init 成功后调用，并绑定当前已生效的运行期配置。若配置缺失、失效或尚未初始化，SDK 必须拒绝创建会话并返回明确错误。
FR-03 权限前置条件 业务层必须在启动识别前完成麦克风权限检查与申请；若权限未就绪导致录音初始化失败，SDK 必须返回明确错误。
FR-04 实时音频上传 识别开始后必须持续采集并分帧发送音频；音频输入格式需在客户端收敛为稳定契约。
FR-05 中间结果输出 库必须能向业务方持续输出 partial result；若后端声明不支持，则通过能力声明明确告知。
FR-06 最终结果输出 正常 stop 收尾路径下，库必须先输出 RecognitionFinal，再输出 SessionCompleted；若本次会话没有可提交文本，则允许仅输出带 no-speech 原因的 SessionCompleted。
FR-07 停止与取消 stop 表示停止继续送音并等待服务端收尾；cancel 表示立即结束并释放本地资源，进入取消终态，不再期待 final 或 completed。
FR-08 状态监听 业务方必须能够监听状态变更、警告、错误与原始诊断事件。
FR-09 扩展能力接入 库必须预留 capability 与 metadata 扩展槽，用于承载时间戳、热词、说话人等可选能力。
FR-10 资源释放 会话完成、取消或失败后必须释放录音资源、传输连接与订阅监听；业务层调用 release 后，SDK 必须清理运行期配置、连接状态和内部缓存。

6. 状态机与生命周期

状态机是库稳定性的核心。Flutter 端必须自建状态机，并把录音、连接和识别会话三层状态统一收敛到可判定的公开行为。

状态 说明 允许的主要动作 退出条件
Idle SDK 已创建但尚未 init，或已经 release。 init 进入 ConfigLoading 或 Failed
ConfigLoading 执行 init，请求并校验运行期配置。 cancel 进入 Ready、Failed 或 Idle
Ready 运行期配置已就绪，可创建会话。 createSession, release 进入 Preparing 或 Idle
Preparing createSession 后校验前置条件、初始化音频、建立连接、完成握手。 cancel 进入 Streaming、Cancelled、Failed 或 Idle
Streaming 持续送音频并接收 partial 结果；本版本把 final 视为 stop 收尾阶段的标准输出。 stop, cancel 进入 Stopping、Cancelled、Failed 或 Idle
Stopping 已停止本地送音，等待服务端 final result 和 completed 确认。 cancel 进入 Completed、Cancelled、Failed 或 Idle
Cancelled 主动取消结束，为终止态，不应再产出 final 或 completed。 release 进入 Idle
Completed 正常收尾完成，为终止态。 release 进入 Idle
Failed 异常结束，为终止态，必须附带错误。 release 进入 Idle
行为约束：本版本不承诺断线后续接原会话。Streaming 或 Stopping 阶段发生网络断开、服务端关闭、超时或录音异常时，必须进入 Failed 并结束当前会话。

7. 接口与数据模型

公开接口层次
对象 职责 核心能力
AsrClient 统一入口 init、createSession、查询能力、release
AsrSession 一次识别任务 start、stop、cancel、dispose、events、state
AudioSource 提供音频帧 start、frames、stop
Transport 实时传输 connect、send、incomingEvents、close
配置上下文要求
  • init 入参必须支持 appName、appVersion、channel、environment 等 app 维度字段。
  • userToken 应作为可选字段存在，仅在后端需要按用户维度决定策略时传入。
  • SDK 应基于 init 入参请求配置接口，并把返回的路由、能力和会话参数作为当前运行期配置，而不是要求业务方手工拼装识别配置对象。
统一事件模型
  • SessionStateChanged：状态流，业务 UI 和控制逻辑的主依据。
  • RecognitionPartial：中间文本结果，允许多次出现。
  • RecognitionFinal：最终文本结果，可带时间戳和扩展 metadata。
  • SessionCompleted：正常收尾事件。
  • SessionCancelled：主动取消终态事件，用于与失败终态区分。
  • SessionFailed：失败终态事件，附带统一错误对象和原始诊断信息。
  • ProviderRawEvent：后端或供应商原始事件透传，不作为主状态机驱动依据。
扩展模型要求
  • 能力声明采用 capability 集合表达，明确本次会话支持 partial、timestamp、speaker、hotword 等扩展项。
  • 额外参数使用 metadata 或 extra 字段承载，不污染稳定核心字段。
  • 后端协议必须带 version 字段，客户端按版本进行兼容判断。

8. 错误处理与恢复策略

本版本允许保留供应商原始错误信息，但仍需提供最小原因分类，以支撑状态机和业务层的统一处理。

错误类别 说明 客户端处理要求
ConfigError 参数缺失、配置非法、版本不兼容。 启动前失败，禁止进入 Streaming。
PermissionError 业务层未完成权限准备，或系统权限状态导致录音初始化失败。 进入 Failed，并向业务层暴露可提示用户补齐权限准备的错误。
AudioError 录音初始化失败、采集异常、格式不支持。 立即结束会话并释放本地音频资源。
TransportError 握手失败、网络中断、连接超时、服务端异常关闭。 进入 Failed；本版本不自动续接原会话。
ProviderError 后端或供应商返回拒绝、额度不足、参数错误。 透传原始 code / message / payload，并附统一原因分类。
StateError 非法状态调用，例如 Completed 后再次 start。 同步返回错误或抛出事件，不允许隐式修复。
必须保留的原始信息
  • 原始错误码。
  • 原始错误消息。
  • 原始诊断 payload。
  • 会话标识与请求标识。
本版本恢复策略
  • 不做原会话无缝续识别。
  • 可选做连接级退避重试，但不得伪装成同一识别会话延续。
  • 业务侧若需重试，应重新创建会话。
  • cancel 属于主动结束，不计入失败重试链路。
完成契约：正常 stop 路径默认顺序为 RecognitionFinal 后 SessionCompleted。若后端明确返回 no-speech 或空结果完成，允许省略 RecognitionFinal，但必须给出完成原因。cancel 路径只允许产出 SessionCancelled，不允许再产出 final 或 completed。

9. 安全与兼容性要求

类别 要求
凭证安全 Flutter 端不得内置供应商密钥；业务层通过 init 传入 appName 与可选 userToken 后，由 SDK 请求配置接口获取运行期配置。配置长期或短期存储不由接口强制规定，但 release 后必须清理 SDK 内部持有数据。
日志安全 默认日志不得打印完整 token、原始 headers、敏感 payload；init 入参中如包含 userToken，必须视为敏感信息处理；调试日志必须可配置开关。
平台兼容 仅以 Android、iOS 为首发支持范围；Web 与桌面不在本版本承诺范围内。
协议兼容 与后端交互的实时协议必须具备 version 和 capability 字段；未知字段不得导致客户端崩溃。
背压控制 音频帧发送必须具备缓存上限与丢弃或阻塞策略，默认发送缓冲上限不超过 64 帧或 1000ms 音频；超限时必须产生 WarningEvent，并按配置策略处理。

10. 测试与验收要求

测试层级 重点覆盖 验收要点
单元测试 状态机、参数校验、错误映射、能力声明、事件顺序。 非法状态调用有明确结果，状态转移无歧义。
集成测试 录音采集、传输层、stop / cancel、超时、断网、服务端关闭，以及不同 appName 上下文下的配置解析。 完整闭环中能够稳定输出 partial 与 final，并在结束时释放资源；cancel 路径只输出取消终态，不再等待 final。
真机测试 业务层权限编排、前后台切换、来电打断、弱网、长时运行。 异常场景统一进入 Failed，不出现卡死、重复回调或内存持续增长。
必须验收的正向场景
  • 首次启动权限通过后可正常开始识别。
  • 业务层未先完成权限准备时，不应直接调用 start。
  • 不同 appName 进入同一 SDK 时，后端能解析到各自对应的会话策略。
  • 识别中可持续收到 partial result。
  • 正常 stop 后默认按顺序收到 RecognitionFinal 和 SessionCompleted。
  • 后端声明不支持 partial 时，会话仍能完成闭环，且客户端不会错误承诺 partial 输出。
  • 会话结束后录音和连接资源均被释放。
必须验收的异常场景
  • 业务层漏做权限准备或系统权限被拒绝时，会话失败且错误明确。
  • userToken 缺失时，若后端策略只按 appName 维度解析，仍可正常完成会话初始化。
  • Streaming 中网络断开后会话失败，不伪装续接。
  • cancel 后只输出 SessionCancelled，且不再产出 final 或 completed。
  • 重复调用 start / stop / cancel / dispose 的返回行为一致且幂等。
  • 未知扩展字段不会导致解析崩溃。
  • 弱网背压时发送缓冲不超过上限，并能观察到 WarningEvent 或既定降级行为。

该时序图描述标准实时识别闭环：创建会话、完成准备、持续送音频、接收中间结果、触发 stop、等待最终结果、完成收尾并释放资源。

客户端主动调用或发送 服务端或内部返回事件
闭环定义：只要进入 stop，客户端必须停止继续送音，并等待服务端按默认顺序返回 RecognitionFinal 与 SessionCompleted；若本次会话没有有效文本，则允许只返回带 no-speech 原因的 SessionCompleted。等待过程中如连接异常，会话进入 Failed，而不是尝试无缝续接。
