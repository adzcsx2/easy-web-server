Flutter 实时 ASR 方案

两套实时语音识别架构方案，分别采用后端中转和客户端适配器两种模式，均面向 Android / iOS 平台的 Flutter 实时流式识别场景。

方案 A

后端中转实时 ASR

Flutter 客户端连接统一后端 WebSocket 协议，模型和供应商切换在服务端完成。SDK 可被多个 App 复用，业务层调用 init 传入 appName 和可选 userToken 后，由 SDK 请求配置接口拿到当前运行期配置；客户端不持有供应商 API Key。

Android / iOS 实时流识别闭环 后端统一协议
  • 统一接入入口，init / createSession / release
  • 本地音频采集与分帧
  • WebSocket 实时会话连接
  • 客户端状态机管理
  • 统一事件输出与扩展能力
后端承担模型适配与密钥管理 查看方案 →
方案 B

客户端适配器实时 ASR

Flutter SDK 持有统一会话内核和多模型适配器。SDK 可被多个 App 复用，业务层调用 init 传入 appName 和可选 userToken 后，由 SDK 请求配置接口获取模型、apiKey 和其他运行期配置，并按配置选择适配器。

Android / iOS 多供应商并存 SDK 内切模型
  • SDK 在 init 阶段请求并消费服务端模型配置
  • 统一模型适配器接口
  • 单页单会话管理
  • 原始事件通道与诊断能力
  • 按适配器契约迭代接入新模型

方案对比

维度 方案 A · 后端中转 方案 B · 客户端适配器
模型逻辑位置 后端统一处理 客户端适配器实现
鉴权方式 后端持有，客户端不可见 前端消费后端下发的模型配置和 apiKey，由后端统一管理密钥，泄露时服务端直接吊销更换
新增供应商 只需更新后端 需新增客户端适配器
配置来源 后端根据 app 参数解析路由与会话参数 后端根据 appName 和可选 userToken 下发模型配置
客户端复杂度 较低 较高
协议灵活性 依赖后端协议版本 客户端可适配多种协议
token 计算 1. 可为不同 App 分配独立 apiKey，通过服务商的 apiKey 粒度 token 消耗统计实现按 App 计量。
2. 可在实际调用链路中记录供应商返回的 usage 字段、结束事件或账单明细；即使供应商没有”已消耗 token 查询接口”，也仍可基于服务端请求响应日志、会话结束回包、账单导出或官方计费规则估算。		 可为不同 App 分配独立 apiKey，通过服务商的 apiKey 粒度 token 消耗统计实现按 App 计量。
优点 客户端接入面更稳定，密钥不出后端，新增供应商通常只改后端；统一审计、限流、计费和问题排查口径更容易收敛；token 统计具备实时链路日志，准确性和实时性更优。 更接近供应商原生能力，试新模型和做差异化能力验证更直接；部分高级特性不必先经过后端协议抽象才能落地。
缺点 后端需要承担实时中转、协议适配、扩缩容和高并发稳定性，服务端建设与运维成本更高。由于增加了一层中间转发与考虑并发压力,稳定性和延迟性上不如方案B 客户端复杂度、发版成本和跨端一致性压力更高；新增供应商通常要补 SDK 适配器，且密钥与配置暴露面更大；token 统计只能依赖服务商账单做事后统计，缺乏实时链路日志。

部署与接入

库开发完成后，统一发布到 Gitee 私有仓库，业务方通过 pubspec.yaml 中的 git 依赖方式引入。以下是两种方案的发布结构、配置方式和接入示例。

发布结构

方案 A · 单库结构
  • asr_flutter — 统一 Flutter SDK：对外暴露 AsrClient、AsrSession、事件模型与音频采集能力
  • 内部按模块划分状态机、传输层、音频采集与测试夹具，但不要求业务方感知内部拆分
方案 B · 单包结构
  • asr_client — 统一客户端 SDK：AsrFacade、ConfigResolver、SessionController、ModelAdapter 及适配器实现
两种方案均以 Gitee 私有仓库为唯一发布源，业务方通过 pubspec.yaml 中的 git 依赖引入，不要复制源码到业务工程中。同一个 SDK 可供多个 App 复用，业务层在调用语音能力前应先调用 SDK 的 init，并传入 appName、环境、渠道等上下文以及可选 userToken；SDK 再请求配置接口获取当前运行期配置。配置长期存储还是短期存储不由 SDK 接口强制规定，但 release 后必须清理运行期持有的配置和资源。

发布方式

方式 适用场景 操作
Gitee 私有仓库 企业内部 Flutter SDK 发布 为仓库打版本 Tag，并在 pubspec.yaml 中通过 git: + ref: 引入

pubspec.yaml 配置

方案 A · 单包依赖
dependencies: asr_flutter: git: url: https://gitee.com/your-org/asr_flutter.git ref: v1.0.0
方案 B · 单包依赖
dependencies: asr_client: git: url: https://gitee.com/your-org/asr_client.git ref: v1.0.0

导入与使用

方案 A · 后端中转
import 'package:asr_flutter/asr_flutter.dart'; // 1. 业务层先完成麦克风权限检查与申请 // await businessPermissionService.ensureMicrophoneGranted(); // 2. 创建客户端（可复用） final asrInitParams = AsrInitParams( appName: 'news_app', appVersion: '1.0.0', channel: 'appstore', environment: 'prod', userToken: currentUserToken, // 可选：仅在需要按用户维度下发配置时传入 ); final client = AsrClient( configEndpoint: 'https://your-backend.example.com/asr/config', ); // 3. 初始化 SDK，内部请求配置接口并缓存当前运行期配置 await client.init(asrInitParams); // 4. 创建识别会话 final session = client.createSession(); // 5. 监听识别事件 session.events.listen((event) { if (event is RecognitionPartial) { /* 更新 UI */ } if (event is RecognitionFinal) { /* 处理结果 */ } if (event is SessionCompleted) { /* 会话结束 */ } }); // 6. 开始识别 await session.start(); // 7. 停止识别 await session.stop(); // 8. 页面退出时释放运行期配置和资源 await client.release();
方案 B · 客户端适配器
import 'package:asr_client/asr_client.dart'; // 1. 业务层先完成麦克风权限检查与申请 // await businessPermissionService.ensureMicrophoneGranted(); // 2. 初始化 SDK，配置加载由 SDK 内部消费 final asrInitParams = AsrInitParams( appName: 'news_app', appVersion: '1.0.0', channel: 'appstore', environment: 'prod', userToken: currentUserToken, // 可选：仅在需要按用户维度返回配置时传入 ); final facade = AsrFacade( configEndpoint: 'https://your-backend.example.com/asr/config', ); // 3. 页面进入时初始化 SDK，内部请求配置接口并切换适配器 await facade.init(asrInitParams); // 4. 创建会话（绑定当前模型配置快照） final session = facade.createSession(); // 5. 监听识别事件 session.events.listen((event) { if (event is RecognitionPartial) { /* 更新 UI */ } if (event is RecognitionFinal) { /* 处理结果 */ } if (event is SessionCompleted) { /* 会话结束 */ } }); // 6. 开始识别 await session.start(); // 7. 停止识别 await session.stop(); // 8. 页面退出时释放运行期配置和资源 await facade.release();
方案 A：AsrClient 可长驻复用，但在调用语音能力前业务层必须先调用 init，并传入 appName 与可选 userToken。SDK 内部请求配置接口获取当前运行期配置；createSession 只消费 init 后已生效的配置，release 用于释放这些配置与资源。
方案 B：AsrFacade 同样通过 init 传入 appName 与可选 userToken，由 SDK 内部请求配置接口并切换适配器，同页只允许一个活动会话，配置变化只影响新会话；release 用于释放运行期配置和适配器资源。

配置差异

维度 方案 A · 后端中转 方案 B · 客户端适配器
后端连接配置 WebSocket endpoint + init 入参 + 会话参数 业务侧只依赖 SDK；业务层先调 init，SDK 再按 appName 和可选 userToken 请求并消费服务端模型配置
鉴权方式 客户端不持有，后端管理 由 SDK 在 init 阶段请求配置接口获取模型配置和 apiKey 供内部消费，密钥由后端统一管理，泄露时可即时吊销
模型选择 后端侧按 app 参数路由，客户端无感知 SDK 内根据服务端按 app 参数返回的配置切换适配器
用户维度 可选在 init 时传入 userToken，由后端决定是否参与会话配置解析 业务层可在 init 时传入 userToken，由后端决定是否返回用户级配置
音频参数 通过统一 SDK 配置采样率、帧长 通过统一 SDK 的 AudioSource 能力管理
扩展能力 capability 集合 + metadata 透传 适配器声明 AdapterCapabilities
两种方案的接入边界都应保持在统一 SDK 内部完成协议与音频链路管理，业务侧只依赖单一 SDK 接口。麦克风权限检查与申请应由业务层先行完成。调用语音方法前，业务层必须先调用 SDK 的 init，并传入 appName、环境、渠道等上下文以及可选 userToken；SDK 再请求配置接口获取当前运行期配置。配置采用长期存储还是短期存储不由 SDK 接口层强制规定，但 release 后必须清理运行期持有的配置和资源。方案 B 当前仍可能由配置接口返回模型配置和 apiKey 供 SDK 内部消费，密钥由后端统一管理，泄露时可即时吊销，但“后端下发长期 apiKey”仍属于当前业务约束下的高风险折中，不应视为长期推荐安全基线。
